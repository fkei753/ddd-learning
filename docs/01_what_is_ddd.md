# ドメイン駆動設計（DDD）とは

## 1. DDDの概要

**ドメイン駆動設計（Domain-Driven Design / DDD）** は、Eric Evans が2003年に著書
*"Domain-Driven Design: Tackling Complexity in the Heart of Software"* で提唱した
ソフトウェア設計アプローチです。

> **核心思想：ソフトウェアの複雑さに立ち向かう最良の方法は、
> そのビジネス領域（ドメイン）を深く理解し、設計に反映させることである。**

---

## 2. なぜ DDD が必要か？

### よくある問題
- コードを読んでも「何をするシステムか」が伝わらない
- 仕様変更のたびに影響範囲が読めない
- 開発者とビジネス担当者が「違う言葉」で話している
- ロジックが UI・DB・外部 API に散在している

### DDD が解決すること
| 問題 | DDD のアプローチ |
|------|----------------|
| 言葉のずれ | **ユビキタス言語** で統一 |
| 巨大すぎるシステム | **境界づけられたコンテキスト** で分割 |
| ロジックの散在 | **ドメイン層** に集約 |
| 変更への弱さ | **集約・エンティティ・値オブジェクト** による明確な境界 |

---

## 3. DDD の二層構造

DDD は大きく **戦略的設計** と **戦術的設計** に分かれます。

```
DDD
├── 戦略的設計（Strategic Design）  ← システム全体の地図を描く
│   ├── ユビキタス言語
│   ├── 境界づけられたコンテキスト（Bounded Context）
│   └── コンテキストマップ
│
└── 戦術的設計（Tactical Design）  ← ドメインモデルの実装パターン
    ├── エンティティ（Entity）
    ├── 値オブジェクト（Value Object）
    ├── 集約（Aggregate）
    ├── ドメインサービス（Domain Service）
    ├── リポジトリ（Repository）
    └── ドメインイベント（Domain Event）
```

---

## 4. ドメインとは

**ドメイン（Domain）** とは、ソフトウェアが解決しようとしている
**現実世界のビジネス領域** のことです。

| システム例 | ドメイン |
|-----------|---------|
| EC サイト | 注文管理、在庫管理、配送管理、決済 |
| 病院システム | 診察、処方、入退院管理 |
| 銀行システム | 口座管理、送金、融資 |

### コアドメイン vs サブドメイン

```
EC サイトのドメイン
├── コアドメイン（Core Domain）    ← 競合優位性の源泉
│   └── 注文管理・レコメンド
│
├── 支援サブドメイン（Supporting）  ← 独自実装が必要
│   └── 在庫管理
│
└── 汎用サブドメイン（Generic）     ← 既製品で代替可能
    └── 認証・決済・メール送信
```
